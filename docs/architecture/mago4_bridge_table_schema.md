# Mago4 ERP Integration: Bridge Table Schema (Tabella Piana)

## 1. Overview
This document outlines the exact data structure required for the "Tabella Piana" (Flat/Bridge Table) used to import orders into the Mago4 ERP system. 

The FastAPI backend must transform the nested JSON extracted by the Gemini AI into a flat list of dictionaries (rows) that strictly adhere to this schema.

## 2. The "Prefix" Rule (Crucial)
Because this is a flat table, order header information and line item information are combined into single rows. Mago4 differentiates them using prefixes:
* **Header Fields (`H_` prefix):** These fields apply to the entire order. Their values **must be repeated** on every row that belongs to the same order.
* **Line Item Fields (No prefix):** These fields apply only to the specific item/variant on that row.

## 3. Data Schema

Below are the required columns to be generated by the backend. The keys in the final output JSON/CSV must match the "Column Name" exactly.

### 3.1 Service Fields (System State)
These fields track the import status.

| Column Name | Data Type | Default Value / Rule | Description |
| :--- | :--- | :--- | :--- |
| `Id` | Integer | (Auto-increment/Unique) | Unique identifier for the row in the bridge table. |
| `CreatedDate` | DateTime | `Now()` | Timestamp of when the AI processed the order. |
| `Processed` | Boolean | `0` | MUST default to `0`. Mago4 will change this to `1` after successful import. |

### 3.2 Header Fields (Testata Ordine - Repeat on every row)
Extracted order-level data mapped to Mago4 header columns.

| Column Name | Data Type | Description |
| :--- | :--- | :--- |
| `H_ExternalOrdNo` | String | The Customer's original Order Number/PO Number. |
| `H_OrderDate` | Date | The date the order was placed. |
| `H_ConfirmedDeliveryDate` | Date | The requested or confirmed delivery date for the order. |
| `H_Notes` | String | General notes or instructions for the entire order. |
| `H_Currency` | String | e.g., "EUR". |

### 3.3 Line Item Fields (Righe Ordine - Unique per row)
Extracted and transcodified item-level data.

| Column Name | Data Type | Description |
| :--- | :--- | :--- |
| `Item` | String | **CRITICAL:** The internal Mago4 Item Code (Result of backend N:1 transcodification). |
| `Description` | String | Description of the specific item/variant. |
| `Qty` | Numeric | Quantity ordered. |
| `UoM` | String | Unit of Measure (e.g., "ML", "MQ", "PZ", "N"). |
| `UnitValue` | Numeric | Unit price of the item. |
| `TaxableAmount`| Numeric | Total row amount (`Qty` * `UnitValue`). |
| `Notes` | String | Specific configuration notes or attributes for this specific line item. |

## 4. Instructions for AI Coding Assistant (Kira)
1.  **Extraction:** When defining the Pydantic schema for Gemini, you can use semantic names (e.g., `customer_order_number`, `unit_price`).
2.  **Transformation:** You MUST include a mapping function in the FastAPI backend that takes the Pydantic output, joins it with the mock master data (Anagrafiche), and outputs a List of Dictionaries.
3.  **Output Mapping:** The keys of the dictionaries in that final list MUST exactly match the `Column Name` values provided in section 3 (e.g., `{"H_ExternalOrdNo": "PO-123", "Item": "SIM-MB-01", "Qty": 5...}`).
4.  **Frontend Display:** The Gradio frontend should display this final list of dictionaries using a `gr.Dataframe` component so the user can verify the Mago4-ready format.
